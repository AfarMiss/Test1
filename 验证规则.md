# design-doc-analysis 验证规则恢复文档

本文件用于下次会话恢复 `list.md` 的 JSON 保护/验证规则。开始处理 `design-doc-analysis` 相关任务前，先读取本文件，再读取 `list.md`、`golden/manifest.json`、`golden/strict-compare-current.json`。Markdown 验收视图规则见 `md-验证规则.md`。

## 权威文件

- `list.md`：JSON 验收状态清单，分为“已验收需保护”“未纳入保护”“排除清单”。
- `all-design-json/`：当前脚本对全部策划案生成的 JSON 输出。
- `all-design-json/summary.json`：全量生成统计。
- `all-design-json/outputs.txt`：全量输出文件列表。
- `golden/manifest.json`：已保护项与 golden JSON 的映射。
- `golden/<safe-name>/extract.json`：用户验收通过后的正确 JSON 基线。
- `golden/strict-compare-current.json`：最近一次保护项严格比对结果。
- `script-backups/`：修改 `extract_excel_design.py` 前的脚本备份目录。

## 基本约束

- 先理解真实需求，禁止直接修改；大任务拆解为可独立完成的小步骤后逐项执行。
- 脚本是通用设计，不能写入文件名、业务词、界面名等特例逻辑。
- 不能依赖固定上下/左右/行列阈值合并，例如固定 `>= 6`、`<= 4`、`vertical_gap <= 40` 这类规则。
- 输出需要保留 `sections` / `blocks` 结构。
- 不输出调试/追踪字段：`cells`、`rows`、`merged_ranges`、`anchor`、`source`、`bounds`、`source_refs`、`reference_sheet`。
- `images` 只保留 `index`、`path`，转换失败时可带 `error`。
- 不再保留“参考”相关功能或输出。

## 用户验收通过时

当用户明确说某个策划案“通过”“验收通过”或“通关”：

1. 从 `list.md` 的“未纳入保护”移到“已验收需保护”。
2. 将当前 `all-design-json/<对应输出目录>/extract.json` 复制为 `golden/<safe-name>/extract.json`。
3. 更新 `golden/manifest.json`，登记该文件名、golden 路径和验收时间。
4. 对全部保护项执行严格比对，更新 `golden/strict-compare-current.json`。
5. 严格比对必须通过后，才算保护状态成立。
6. 如果只是新增保护项，没有修改脚本，不需要重新生成全部 JSON。

## 用户撤回或指出已保护项问题时

当用户指出已保护项还有问题：

1. 先不要修改脚本，先理解用户指出的真实结构问题。
2. 将该项视为待修正；必要时从 `list.md` 的“已验收需保护”移回“未纳入保护”。
3. 同步移除或作废 `golden/manifest.json` 中对应项，避免错误 golden 阻塞后续验证。
4. 修正方向必须先通过临时输出或 dry-run 验证目标文件对上，再给出方案。
5. 用户确认方案后，才允许修改 `extract_excel_design.py`。

## 修改脚本前

每次修改 `C:/Users/admin/.agents/skills/design-doc-analysis/scripts/extract_excel_design.py` 前：

1. 先备份到 `.codex/design-doc-analysis/script-backups/`。
2. 备份文件名需要包含时间戳，便于还原。
3. 说明将修改的通用规则，不使用业务特例。

## 修改脚本后

只要 `extract_excel_design.py` 被修改：

1. 必须重新输出全部 JSON 到 `.codex/design-doc-analysis/all-design-json/`。
2. 必须更新 `summary.json` 和 `outputs.txt`。
3. 全量输出顺序尽量保持历史稳定，避免保护项 `images.path` 因目录编号变化产生无意义 diff。
4. 必须对所有 `golden/manifest.json` 中的保护项执行严格 JSON 比对。
5. `golden/strict-compare-current.json` 必须显示所有保护项通过。
6. 如果保护项比对失败，不能直接交付；需要继续修正或回滚脚本。

## 严格比对标准

- 保护项以 `golden/<safe-name>/extract.json` 为唯一正确基线。
- 当前输出与 golden 必须完整 JSON 相等。
- 比对不是只看标题、数量或局部文本，而是整个 JSON 结构和值。
- 允许比对工具做稳定化读取，例如排序 key；但不能忽略真实字段差异。
- 同时检查禁止输出的调试字段，避免 `cells`、`anchor` 等重新进入结果。

## 排除项处理

- `list.md` 的“排除清单”用于记录不参与当前主需求提取的文件。
- 没有命中主需求 sheet 的美术需求文档可以留在排除清单。
- 当前脚本不支持 `.xls`，对应文件可留在排除清单。

## 恢复检查顺序

下次恢复时按以下顺序检查：

1. 读取本文件。
2. 读取 `list.md`，确认保护项、未保护项、排除项。
3. 读取 `golden/manifest.json`，确认保护项与 golden 数量一致。
4. 查看 `golden/strict-compare-current.json`，确认最近一次严格比对是否通过。
5. 如果要修改脚本，先备份，再按“修改脚本后”流程全量生成并严格验证。
